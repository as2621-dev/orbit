# Phase 4: X half — Following loader, SearchTimeline delta, classify into the shared pipeline

**Milestone:** M2 — X half
**Status:** Not started
**Estimated effort:** L

## Goal
Orbit loads the user's X following list via a newly-added `Following` GraphQL op on the vendored bird client (cookie auth, cursor pagination, queryId live-refresh), pulls each handle's recent tweets via `from:handle` SearchTimeline as a paced, rate-limit-aware delta, and feeds classified X items into the **same** ranking and render path built in M1 — producing one unified digest spanning YouTube and X.

## Resolved open questions folded into this phase
- **Following-op coverage (master-plan Q4) — RESOLVED: ABSENT.** Inspection found the vendored `bird-search` client exposes only `SearchTimeline` as an executable op. `Following` exists merely as a queryId string (`query-ids.json` + `FALLBACK_QUERY_IDS`) plus a `buildFollowingFeatures()` scaffold in `twitter-client-features.js:264` — there is NO `withFollowing` mixin, no client method, no CLI surface, and `bird_x.py` never references it. Therefore **M2 MUST extend the client** — this is Sub-phase 1 below (a real `withFollowing` mixin + cursor pagination + CLI flag + Python wrapper).
- **depth cap + handle-rotation fairness for X (master-plan Q5) — RESOLVED.** The vendor client has NO rate-limit pacing (the `paginate-cursor.js` `pageDelayMs` exists but is dead/unused; `searchPaged` fires pages back-to-back). `bird_x.py` fans out up to 5 concurrent Node subprocesses with no fairness. Policy pinned in Sub-phase 3: **deterministic round-robin handle rotation keyed on the run's day-ordinal** so over successive days every followed handle gets deep-pulled even when the follow count exceeds the per-run deep-pull budget; per-run budget derived from `DEPTH_CONFIG` (`quick:12, default:30, deep:60`); add an inter-request delay (wire the unused `pageDelayMs`) and keep the bounded ThreadPool but cap concurrency conservatively to respect X's ToS-gray posture.
- **queryId refresh** is reused verbatim (`runtime-query-ids.js` bundle-scrape, 24h TTL cache, refresh-on-404 in `searchPaged` / generic `withRefreshedQueryIdsOn404`). The new Following op plugs into the same refresh wrapper.

## Sub-phases

### Sub-phase 1: Extend the vendored bird client with the `Following` op ⚠ irreversible
- **Files touched:** `skills/orbit/scripts/lib/vendor/bird-search/lib/twitter-client-following.js` (new mixin), `skills/orbit/scripts/lib/vendor/bird-search/bird-search.mjs` (compose mixin + CLI flag), `skills/orbit/scripts/lib/vendor/bird-search/lib/twitter-client-base.js` (add `getFollowingQueryIds` helper if needed)
- **What ships:** A `withFollowing(Base)` mixin adding a `following(screenName, {count})` method that builds the GraphQL request using the existing `Following` queryId + `buildFollowingFeatures()`, walks cursor pagination (reusing the `paginate-cursor.js` helper with `pageDelayMs`), and extracts followed-account entries (`creator_handle`, `display_name`, `rest_id`) from the timeline instructions. Composed into the CLI as `bird-search --following <screen_name> [--json]`. Plugs into the existing `withRefreshedQueryIdsOn404` refresh-on-404 path. This is the one piece the reference does NOT have. Vendor code is EXTENDED, never rewritten (conventions rule).
- **Definition of done:** `node skills/orbit/scripts/lib/vendor/bird-search/bird-search.mjs --help` (or `--following` invoked with a stub) shows the new flag; a Node-level unit test (or a fixture-driven harness) feeds a canned `Following` GraphQL JSON response and asserts the mixin parses out the expected `creator_handle`s and follows the `next_cursor` across two pages (cursor-pagination intent — a single page must not silently truncate a long follow list). No live X call. queryId resolution is mocked to a fixture value.
- **Dependencies:** none

### Sub-phase 2: Python wrapper for Following — X source loading (Stage 0)
- **Files touched:** `skills/orbit/scripts/lib/bird_x.py`
- **What ships:** `load_x_following(cookie_source) -> list[Follow]` that resolves cookies (priority: CLI args → env `AUTH_TOKEN`/`CT0` → browser cookie store, exactly the `twitter-client-base.js` recipe), shells out to `bird-search.mjs --following <me> --json` via `lib/subproc`, parses the JSON into typed `Follow` records, and `persist_following(follows)` upserts each into `sources` with `platform="x"`, `external_id=creator_handle`, `category="signal"` default, `last_refreshed_at=now`. Cookie values are NEVER logged (redaction at the boundary). Expired/absent cookies raise a clear, actionable auth error pointing at README §8.6 (not a silent death).
- **Definition of done:** A test mocks the `bird-search.mjs` subprocess to return a canned following JSON and asserts `load_x_following` returns the right `creator_handle`s and `persist_following` makes them queryable via `store.list_sources(platform="x")`. A failure test makes the subprocess return an auth-failure signal and asserts the raised error mentions re-logging-in / README troubleshooting (loud auth failure). A test asserts no cookie/`auth_token`/`ct0` value appears in any captured log line (security invariant). No live X call.
- **Dependencies:** Sub-phase 1

### Sub-phase 3: Per-handle SearchTimeline delta with pacing + rotation (Stage 1)
- **Files touched:** `skills/orbit/scripts/lib/bird_x.py`
- **What ships:** `fetch_new_tweets(sources, depth, run_day_ordinal) -> list[Tweet]` that, for the X sources, selects the deep-pull set by **deterministic round-robin rotation** (handles ordered stably; window of size `DEPTH_CONFIG[depth]` rotated by `run_day_ordinal % len(handles)`) so coverage rotates across days for large follow counts; for each selected handle runs `from:handle` SearchTimeline (lifted, cursor-paginated, refresh-on-404), filters tweets whose `tweet_id` is NOT in `store.get_seen_ids`, and applies an inter-request delay + bounded concurrency to pace conservatively (ToS-gray posture, brief §8.5). Returns new `Tweet`s (text, tweet_id, engagement counts, created_at, handle); `store.mark_seen` only after success.
- **Definition of done:** A test mocks SearchTimeline output, pre-seeds `seen` with some `tweet_id`s, and asserts only unseen tweets are returned (delta intent). A test with a follow list larger than the per-run budget asserts that across two consecutive `run_day_ordinal` values the UNION of deep-pulled handles strictly grows toward full coverage (rotation-fairness intent — a high-follow user is not permanently starved; this is the resolved Q5 policy, tested, not just asserted in prose). A test asserts the inter-request delay is invoked between handles (pacing). No live X call.
- **Dependencies:** Sub-phase 2

### Sub-phase 4: Feed X items into the shared classify + pipeline path
- **Files touched:** `skills/orbit/scripts/lib/classify.py`, `skills/orbit/scripts/orbit.py`
- **What ships:** `classify.classify_item` is generalized so an X `Tweet` (text-only, no transcript) classifies on the same two axes with the same channel(handle)-level prior — the existing classify path accepts both YouTube uploads and X tweets via a normalized item shape (no separate X classifier; reuse, don't fork). `orbit.py` Stage 0/1 now load BOTH sources (YouTube subs from M1 + X following) and merge YouTube uploads + X tweets into one item stream before classify → (M1's) score → tier → render. The unified stream flows into the same `rerank.py`/`density.py`/`render.py` from Phase 3 unchanged — X tweets appear as cards in the digest alongside videos.
- **Definition of done:** A test feeds a normalized X `Tweet` through `classify_item` with a mocked LLM and asserts it produces a valid two-axis `Classification` persisted to `store.classifications` (shared-path intent — no X-specific classify table or function). An `orbit.py` integration test with mocked YouTube AND X sources asserts the rendered HTML (from Phase 3's `render_digest_html`) contains BOTH a YouTube `watch?v=` card and an X tweet card in the same digest (unified-digest intent, the M2 deliverable). LLM and both subprocess boundaries mocked.
- **Dependencies:** Sub-phase 3; and (cross-phase) Phases 1-3 complete — consumes M1's store, classify, rerank, density, render.

## Phase-level definition of done
`pytest tests/` passes. Given mocked cookies, a full run loads the X following list (via the new `Following` op), deltas each rotated handle's `from:handle` tweets, classifies them on the shared two-axis path, and renders ONE HTML digest containing both YouTube video cards (with deep-links) and X tweet cards — the unified M2 deliverable. The new bird-client Following op parses a multi-page follow list under the existing queryId-refresh machinery, and the handle-rotation policy demonstrably widens coverage across day-ordinals.

## Out of scope
- No overlap/clustering between X and YouTube items (M3 — they coexist as separate cards here).
- No internal/external trending or scoop detection on X velocity (M3).
- No delivery (iMessage/WhatsApp) or `--setup` (M4).
- No rewrite of the vendored client — only the additive `withFollowing` extension.
- No live X/YouTube/LLM calls in tests — all boundaries mocked with fixtures.

## Open questions
- The exact GraphQL response shape for `Following` (instruction/entry path to extract handles) must be confirmed against a real fixture during Sub-phase 1; the `buildFollowingFeatures()` scaffold and queryId exist, but the entry-extraction path is built from the SearchTimeline analog. Captured as a Sub-phase 1 fixture task — not blocking the plan.
- Conservative pacing constants (inter-request delay, max concurrency) are first-cut; the README risk disclosure (M4 §8.5) will document them and the maintainer tunes against real rate-limit behavior. Not blocking.
